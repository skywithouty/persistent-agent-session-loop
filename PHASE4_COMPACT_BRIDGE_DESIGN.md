# Phase 4 Compact Bridge Design

**DEPRECATED — HISTORICAL REFERENCE ONLY (round 16, reinforced round 17):** The custom Post-Task Compact Gate and ALL context monitoring infrastructure described in this document are deprecated. Do NOT implement any policy, script, or workflow described below. The project relies exclusively on Claude Code's default auto-compact behavior. Scripts (`claude-statusline.ps1`, `check-compact-gate.ps1`) are legacy artifacts — they are NOT copied by the Phase 5 installer, NOT required by doctor validation, and NOT referenced by active loop documentation. This document exists solely as an archive of design decisions made during Phase 4.

## Purpose (Historical)

Define how the persistent session loop should enforce mandatory `/compact` when real Claude Code context usage reaches the threshold, without assuming the bridge must own the Claude Code process.

This is a design document. It does not implement anything.

## Terminology

### Statusline Monitor

The PowerShell script at `.session-loop/context/claude-statusline.ps1`. It is a **passive** data writer: Claude Code calls it on each statusline tick, it receives JSON via stdin, and it writes `claude-status.json` to disk. It never decides anything — it only records.

### Observer

A process or loop that **reads** `claude-status.json` and decides whether compaction is needed. The observer is the decision point. Currently, the observer role is split between:
- Claude's `/loop` (which reads `used_percentage` and writes a recommendation in `## Context Usage`),
- Codex (which reads the report and could act on the recommendation).

Neither currently enforces compaction.

### Bridge

A process that **owns** the Claude Code lifecycle: launches it with known flags, monitors its output, watches context usage, and can inject commands into the session. The original spec (`SESSION_LOOP_SPEC.md`) assumes the bridge launches Claude Code itself. This conflicts with the current reality: Claude Code is **already running in Cursor**, and the bridge did not start it.

### Injector

The mechanism that actually sends `/compact` (or an equivalent `compact()` API call) into a running Claude Code session. Injection strategies:
- The `/loop` cron handler itself could invoke `/compact` via the Skill tool if context exceeds threshold.
- A bridge that owns the process could send commands through the same channel it uses to launch.
- A terminal automation tool could type `/compact` into the Cursor terminal pane.

### Already-Running Claude Code Session

The current reality: the user opens Cursor, starts a terminal, runs `claude`, and leaves it running. No bridge launched it. No external process owns it. The session's only external interfaces are:
- The `/loop` cron (internal to the session),
- The statusline script (called by the session),
- Files on disk (`STATE.json`, `claude-status.json`, task files).

Any Phase 4 design must start from this constraint.

## The Stale/Mock Status Problem

### Current State

The `claude-status.json` on disk was generated by piping `sample-statusline-input.json` through the statusline script during Phase 3 verification. The `used_percentage = 22.5` is a fixed mock value, not a live reading from the actual Claude Code session.

### Required Fixes

1. **Distinguish real from mock status.** The statusline script must record a `source` field:
   - `"live"` — real Claude Code statusline data with usable `context_window.used_percentage`.
   - `"mock"` — sample/test data marked with `is_mock: true` or `source: "mock"` in the input JSON.
   - `"unknown"` — valid JSON received but no `context_window.used_percentage` present; not usable for compaction decisions.
   - `"error"` — empty stdin or malformed JSON.

2. **Timestamp age check.** Before trusting `used_percentage`, compare the `ts` field against the current time. If the timestamp is older than a configurable freshness window (default: 2 loop intervals, ~10 minutes), treat the status as stale.

3. **Status source metadata.** Extend `claude-status.json` output to include:
   ```json
   {
     "ts": "2026-07-20T22:10:00+08:00",
     "used_percentage": 45.0,
     "model": "Claude Sonnet 4.6",
     "cwd": "...",
     "source": "live",
     "session_id": "abc123"
   }
   ```

4. **What to do when status is missing, stale, or mock:**
   - If `claude-status.json` is missing: the statusline is not configured. Treat context as **unknown**. Block before the next substantial task: set `next_actor = "user"` with `stop_reason = "status_missing"` and a message directing the user to configure the statusline or explicitly override.
   - If `claude-status.json` is present but `ts` is older than the freshness window: treat as stale. Treat context as **unknown**. Block before the next substantial task: set `next_actor = "user"` with `stop_reason = "status_stale"`.
   - If `source` is `"mock"`: treat context as **unknown** (not safe-low). Block before the next substantial task: set `next_actor = "user"` with `stop_reason = "status_mock"`. Mock data must never substitute for live context when deciding whether compaction is needed.
   - **User override for all blocks:** The user may manually set `next_actor` back to `"codex"` (or `"claude"`) to override any of these blocks, acknowledging they accept the risk of unknown context.

## Post-Task Compact Gate (Phase 4a Policy)

The user's requirement: real context usage at or above threshold means compaction is **mandatory**, not advisory.

**This policy is named the Post-Task Compact Gate. It MUST never interrupt a task in progress.** Compaction is checked exactly once per round: after the current task completes and before the loop hands off to the next actor or accepts another substantial task.

### Policy Rules

| Condition | Action |
|---|---|
| Status missing, stale, mock, unknown, or error | Treat context as **unknown**. Block before the next substantial task: set `next_actor = "user"`, `stop_reason = "status_missing"` / `"status_stale"` / `"status_mock"`. Do not execute further substantial tasks until live status is available or the user explicitly overrides. |
| `used_percentage < threshold` (live, fresh) | Continue normally. Report the percentage. |
| `used_percentage >= threshold` (live, fresh) | Compaction is **mandatory** before the next substantial task. |
| Automatic injection available | Trigger compaction, verify context dropped, then continue. |
| Automatic injection not available | Set `next_actor = "user"` with `stop_reason = "context_at_threshold"` and a clear message: "Context at X%. Run /compact manually, then set next_actor back to codex." Do not execute further tasks until confirmed. |

### Policy Rules for the /loop

After every task completion, before handing state to Codex:

1. Read `claude-status.json`.
2. Validate freshness and source.
3. If status is mock, stale, missing, unknown, or error: do NOT set `next_actor = "codex"`. Instead set `next_actor = "user"` with `stop_reason = "status_mock"` / `"status_stale"` / `"status_missing"`. Treat unknown context as blocking before the next substantial task.
4. If live, fresh, and `used_percentage >= threshold`: do NOT set `next_actor = "codex"`. Instead set `next_actor = "user"` with `stop_reason = "context_at_threshold"`.
5. Only set `next_actor = "codex"` when context is confirmed: live, fresh, and `used_percentage < threshold`.

## Implementation Options Comparison

### Option A: Manual Gate Only

**Description:** The `/loop` and Codex both check `## Context Usage` but take no automatic action. If context is high, the report says so. A human reads it and types `/compact`.

**Advantages:**
- Zero new code.
- Works today with no external dependencies.
- Cannot break anything.

**Disadvantages:**
- Does not enforce the user's requirement.
- Human forgets → context overflows → session quality degrades.
- Not a "loop"; it's a human-in-the-loop gate.

**Verdict:** Rejected. Does not meet the Phase 4 goal.

### Option B: Observer + User Gate (Recommended First Step)

**Description:** Enhance the current `/loop` to act as the observer. After each task:
1. Read `claude-status.json`.
2. Validate freshness and source.
3. If live + fresh + `used_percentage >= threshold`: set `next_actor = "user"`, supply a clear `/compact` instruction in the report, and refuse to pick up the next task until context drops.
4. If mock, stale, or missing: treat context as unknown, set `next_actor = "user"` with the appropriate `stop_reason`, and block further substantial tasks until live status is available or the user overrides.

**Advantages:**
- No bridge process. No new process to launch or supervise.
- Uses only the existing `/loop` mechanism and file-based state.
- Enforces the strict policy: the loop blocks itself.
- Works with the current "Claude already running in Cursor" model.
- The user can run `/compact` manually and then set `next_actor = "codex"`.

**Disadvantages:**
- Still requires human action to run `/compact` (but the loop enforces it by blocking).
- Manual recovery is verified by Phase 4a: `check-compact-gate.ps1 -Apply` reads the live `claude-status.json` after manual `/compact` and verifies `used_percentage` dropped below threshold before restoring `next_actor = "codex"`.
- Depends on `/loop` itself not being degraded by high context (if the loop tick skips due to context pressure, the observer is blind).

**Verdict:** Implemented as Phase 4a, live-validated in round 12. It enforces the policy with minimal new machinery.

### Option C: Bridge-Launched Claude Session with Injection

**Description:** A separate process (PowerShell, Node, or Python script) that:
1. Launches `claude` with known flags (path, model, working directory).
2. Reads `claude-status.json` on a polling interval.
3. When threshold is crossed: injects `/compact` into the Claude session's stdin or via a control channel.
4. Logs everything.

**Advantages:**
- Full automation: no human needed for compaction.
- Bridge can verify compaction happened (check context after injection).
- Bridge can restart Claude if the session crashes.

**Disadvantages:**
- Requires the bridge to own the Claude process — conflicts with "already running in Cursor."
- Terminal I/O for CLI tools is fragile across platforms (Windows conhost vs. Windows Terminal vs. Cursor's embedded terminal).
- If the bridge launches Claude, the user loses the visual Cursor terminal experience (unless the bridge attaches to Cursor's terminal, which is even more fragile).
- Significantly more complex than Option B.
- A bridge crash could orphan the Claude process or lose state.

**Verdict:** Do not build yet. Revisit only if Option B proves insufficient in practice.

### Option D: Terminal Automation Injector (Cursor-Specific)

**Description:** A script that sends keystrokes to the Cursor terminal window to type `/compact` and press Enter. On Windows, this could use the Windows Accessibility API or SendKeys.

**Advantages:**
- Works with the "already running in Cursor" model.

**Disadvantages:**
- Extremely fragile: window title changes, focus changes, multiple Cursor windows, DPI scaling.
- Security: sending keystrokes to arbitrary windows is a malware pattern. Antivirus may block it.
- Cursor may not expose a stable window handle or automation interface.
- Not portable across OS or terminal emulators.

**Verdict:** Rejected for the MVP. Too fragile and platform-specific. Revisit only if Cursor or Claude Code exposes a formal automation API.

## Recommended Implementation Sequence

**All steps follow the Post-Task Compact Gate rule:** compaction is checked only after a task completes, never mid-task.

1. **Phase 4a (Observer + User Gate) — DONE (live-validated):** The `/loop` prompt and Claude's task-completion routine now enforce the Post-Task Compact Gate. After each task:
   - Live context at/above threshold → block with `next_actor = "user"` and `/compact` instruction.
   - Mock/stale/missing/unknown/error status → block with `next_actor = "user"` and appropriate `stop_reason`.
   - Manual `/compact` recovery is verified by re-running `check-compact-gate.ps1 -Apply`, which reads the live `claude-status.json` and only restores `next_actor = "codex"` when `used_percentage` drops below threshold.
   This required:
   - Adding `source`, `is_mock`, `input_has_context_window` to the statusline script.
   - Updating the `/loop` prompt to enforce the strict threshold policy.
   - Creating `check-compact-gate.ps1` — a deterministic PowerShell checker with `-Apply` switch.
   - Updating `START_HERE_FOR_CLAUDE.md` and `CODEX_REVIEW_RHYTHM.md` with the new enforcement rules.

2. **Phase 4b (Verification) — DEFERRED:** Manual recovery verification is already covered by Phase 4a: `check-compact-gate.ps1 -Apply` verifies the drop. Automatic bridge-based verification remains deferred to Phase 4c.

3. **Phase 4c (Bridge Prototype, only if needed):** If the manual-gate approach proves too slow or the user wants full automation, prototype a bridge that launches Claude Code with known flags and injects `/compact` via stdin. This should be a separate project that can be opted into, not a replacement for the Cursor-terminal workflow.

## Summary

| Question | Answer |
|---|---|
| What does "launch/supervise" mean when Claude is already running? | For Phase 4a, it means nothing — we don't launch anything. We observe from within the session via `/loop`. |
| How do we prevent mock status from triggering or suppressing compaction? | Source metadata (`source: "live"` vs `"mock"`) + `is_mock` flag + `input_has_context_window` + timestamp freshness check. Mock/stale/missing/unknown/error status is treated as **unknown** and blocks further substantial tasks until live status is available or the user overrides. |
| What happens when context hits threshold? | The `/loop` blocks after the current task completes: sets `next_actor = "user"` and refuses further substantial tasks until `/compact` runs and context drops. |
| When is compaction checked? | **Only after a task completes**, never mid-task (Post-Task Compact Gate). |
| Do we need a bridge process now? | No. Option B (observer + user gate) enforces the policy without a separate process. |
